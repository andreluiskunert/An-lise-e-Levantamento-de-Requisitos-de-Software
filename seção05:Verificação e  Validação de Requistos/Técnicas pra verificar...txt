Primeiro, nós temos uma técnica chamada Análise de requisitos por pares em inglês.

Vocês vão ouvir muito a sigla ou a nomeclatura chamada Peer review.

O que que é isso e pra que serve?

Vamos lá.

Revisão por pares é uma das técnicas mais utilizadas para verificar a qualidade dos requisitos.

Nesse método, os requisitos são submetidos a uma equipe de especialistas, que inclui desenvolvedores,

gerentes de projetos, quando possível, os próprios stakeholders, ou seja, os próprios interessados,

os clientes, os usuários finais e etc.

Cada membro da equipe revisa os requisitos, buscando por inconsistências, ambiguidades, falta de

detalhes e conflitos entre eles.

Essa abordagem colaborativa permite identificar problemas em uma fase inicial e reconhecer os requisitos

com diversos pontos de vista.

Então, quando você levantar os requisitos, é interessante sempre fazer o peer review, que é entregar

esses requisitos levantados a diferentes pessoas envolvidas nos projetos.

Então, muitas vezes é a própria equipe no qual você está atuando.

É a sua equipe de desenvolvimento e sua equipe de gestão de projetos e propriamente até mesmo a sua

equipe de gestão de produtos.

E eles vão fazer essas verificações, essas análises.

E aí, se necessário, você também pode pedir para eles ter que holders externos também fazer essas

análises.

Agora quanto ao segundo, é uma prototipagem rápida.

Prototipagem rápida é uma técnica que envolve a criação rápida de protótipos ou modelos de partes do

sistema ou do produto com base nos requisitos coletados.

Esses protótipos são apresentados aos stakeholders para a validação e feedback.

A partir desses protótipos, é possível obter insights valiosos sobre a adequação dos requisitos às

necessidades reais dos usuários, além de identificar novos requisitos ou ajustar os requisitos já existentes.

Bom, aqui, quanto à prototipagem, talvez vocês nunca ouviram a palavra prototipagem rápida dentro

É um MVP que é o mínimo produto viável, que também às vezes nem tem tanta relação com o contexto atual

Porque um MVP já é um produto que está funcionando, só que com os mínimos de requisitos necessários

ou com aquilo que é premissa básica para ele estar funcionando.

A terceira técnica seria a matriz de rastreabilidade de requisitos.

O que faz e para que serve?

Vamos lá.

A Matriz de Rastreabilidade é uma ferramenta que relaciona requisitos com outros artefatos do projeto,

como casos de testes, componentes de software e até mesmos os objetivos de negócio.

Ela permite verificar se todos os requisitos têm uma origem e justificativa bem definida, além de garantir

que cada requisito seja testável e que haja cobertura adequada durante o processo de desenvolvimento

e de teste.

Então aqui.

Na verdade, a matriz de rastreabilidade.

De uma forma bem resumida, ela é um depara nos requisitos que foram levantados de tudo aquilo que é

necessário, que você mapeou com outras áreas ou outro com outros contextos que envolvem o projeto.

Aí, no caso, entra muito em relação a regra de negócio para saber se os requisitos tem relação com

regras de negócio, entrar muito em relação aos testes para entender se os requisitos e os testes que

estão sendo feitos tem relação aos componentes que foram desenvolvidos, as funcionalidades que estão

no ar no momento e etc.

Já a nossa quarta técnica é check list de qualidade de requisitos.

Os check list são listas de verificação que contêm critérios específicos para avaliar a qualidade dos

requisitos.

Esses critérios podem incluir clareza, consistência, rastreabilidade, testabilidade, entre outros.

Os check lists oferecem uma abordagem estruturada e sistêmica para revisar os requisitos, tornando

o processo mais abrangente e evitando que questões importantes sejam esquecidas.

permitir novos acessos ou novos recursos a partir de usuários que já estão existentes ou restringir

esses usuários.
